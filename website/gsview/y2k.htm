GSview Year 2000 compliance

GSview is provided with absolutely no warranty of any kind.

The current version of GSview (2.7) uses dates in four places:

  1. When GSview opens a file, it saves the file length and modification time, returned by the C library function fstat(). Whenever GSview reopens this file, it performs an equality test on this information to determine if the file has changed. Times are stored in Unix format.
  2. When writing EPS files, GSview inserts the %%CreationDate: comment using the standard C run time library functions time() and ctime().
  3. When writing TIFF files, GSview inserts the date using the standard C run time library time() and localtime(), The date is written as a text string in the form YYYY:MM:DD HH:MM:SS, as per the TIFF specification.
  4. In beta versions only, GSview compares the system time with an expiry date, and refuses to operated after this date. This does not apply to release versions of GSview.
The Unix time format stores dates as seconds since 1 January 1970 in variables of type time_t. On most systems today, including those used by Ghostgum Software Pty Ltd, this is a 32-bit signed number which will roll over during the year 2038. It is hoped that by the time 2038 arrives, time_t will be 64-bits or larger.

The use of dates by GSview depends on the C run time library, over which we have no control.
